## Goal

Expose two HTTP entry points that trigger trend ingestion:

1. **Vercel Cron** → `POST /api/cron/trend-ingest` — runs every 6 hours, validates `x-cron-secret`, creates a `trend_ingestion_runs` row with `source='vercel-cron'`, enqueues the `ingest-trends` BullMQ job DIRECTLY (no `jobs` DB row, per rule 14), returns 202.
2. **Manual refresh** → `POST /api/trends/refresh` — session-gated, rate-limited 1/min per user, creates a `trend_ingestion_runs` row with `source='manual-admin'`, enqueues the same BullMQ job, returns 202 with the new run id.

Per rule 11, the cron route validates `x-cron-secret` against `env.CRON_SECRET` and returns 401 on any mismatch. Per rule 14, neither entry point writes to the `jobs` DB table — only `trend_ingestion_runs`.

### Task 1: In-memory rate limiter helper

**File**: `src/lib/trends/rate-limit.ts` (NEW)

**Action**: a dead-simple per-user rate limiter. Tracks last-call timestamps in an in-memory `Map<string, number>`. Cleans stale entries lazily on each call. Window: 60s. Per rule 16, in-memory is acceptable for Phase 9.

```ts
/**
 * Phase 9 rule 16 — per-user manual refresh rate limiter.
 *
 * Window: 60 seconds. Key: arbitrary string (typically session.user.id).
 * Storage: in-memory Map. This is acceptable for Phase 9 because
 * manual refresh is a low-traffic admin-grade action. When the app
 * scales beyond a single Vercel instance, replace this with a Redis
 * INCR + EXPIRE pattern — the interface below is intentionally a
 * single function so swapping is mechanical.
 */

const WINDOW_MS = 60 * 1000;
const lastCall = new Map<string, number>();

/**
 * Attempt to acquire a rate-limit token for the given key. Returns
 *   { allowed: true }                  — caller may proceed
 *   { allowed: false, retryAfterMs }   — caller should 429
 */
export function tryAcquireRefreshToken(
  key: string,
  now: number = Date.now()
): { allowed: true } | { allowed: false; retryAfterMs: number } {
  // Lazy prune: if the map gets hot, sweep stale entries.
  if (lastCall.size > 1024) {
    for (const [k, ts] of lastCall) {
      if (now - ts > WINDOW_MS) lastCall.delete(k);
    }
  }

  const prev = lastCall.get(key);
  if (prev !== undefined && now - prev < WINDOW_MS) {
    return { allowed: false, retryAfterMs: WINDOW_MS - (now - prev) };
  }
  lastCall.set(key, now);
  return { allowed: true };
}

/** Test-only helper to reset the in-memory map. */
export function __resetRateLimitForTest() {
  lastCall.clear();
}
```

### Task 2: Cron route

**File**: `src/app/api/cron/trend-ingest/route.ts` (NEW)

**Action**: the entire route is a single POST handler. It:

1. Reads `x-cron-secret` from the incoming request headers and compares to `env.CRON_SECRET`. Returns 401 on any mismatch (missing, empty, wrong value). Rule 11.
2. Creates a `trend_ingestion_runs` row with `source='vercel-cron'`, `status='pending'`, `regionCode='KR'`.
3. Enqueues the `ingest-trends` BullMQ job directly via `getQueue().add(...)`. Per rule 14, NO `jobs` DB row is created. The payload carries `{ ingestionRunId, regionCode }`.
4. Returns `202 Accepted` with `{ ingestionRunId, source: 'vercel-cron' }`.

```ts
import { NextRequest, NextResponse } from "next/server";
import { db } from "@/lib/db";
import { trendIngestionRuns } from "@/lib/db/schema";
import { getQueue } from "@/lib/queue";
import { env } from "@/lib/env";

/**
 * Phase 9 R-02 + rule 11.
 *
 * Vercel Cron → POST /api/cron/trend-ingest every 6h (see vercel.json).
 * Validates x-cron-secret, creates a trend_ingestion_runs row with
 * source='vercel-cron', enqueues the BullMQ ingest-trends job, returns 202.
 *
 * Rule 14: does NOT write to the `jobs` DB table. Cron runs are
 * tracked exclusively in trend_ingestion_runs.
 */
export async function POST(request: NextRequest) {
  // Rule 11: secret validation is the FIRST thing this route does.
  const provided = request.headers.get("x-cron-secret");
  if (!provided || provided !== env.CRON_SECRET) {
    return NextResponse.json({ error: "unauthorized" }, { status: 401 });
  }

  const [run] = await db
    .insert(trendIngestionRuns)
    .values({
      regionCode: "KR",
      status: "pending",
      source: "vercel-cron",
    })
    .returning({ id: trendIngestionRuns.id });

  await getQueue().add("ingest-trends", {
    payload: {
      ingestionRunId: run.id,
      regionCode: "KR",
    },
    // No jobId — rule 14. The worker handler receives payload only.
  });

  return NextResponse.json(
    { ingestionRunId: run.id, source: "vercel-cron" },
    { status: 202 }
  );
}
```

### Task 3: Manual refresh route

**File**: `src/app/api/trends/refresh/route.ts` (NEW)

**Action**: session-gated, rate-limited. Logic:

1. `getServerSession()` → 401 if no user.
2. `tryAcquireRefreshToken(session.user.id)` → 429 + `Retry-After` header if not allowed.
3. Insert `trend_ingestion_runs` row with `source='manual-admin'`.
4. Enqueue BullMQ `ingest-trends`.
5. Return `202` with `{ ingestionRunId, source: 'manual-admin' }`.

```ts
import { NextResponse } from "next/server";
import { db } from "@/lib/db";
import { trendIngestionRuns } from "@/lib/db/schema";
import { getQueue } from "@/lib/queue";
import { getServerSession } from "@/lib/auth/get-session";
import { tryAcquireRefreshToken } from "@/lib/trends/rate-limit";

/**
 * Phase 9 R-08 — manual refresh button on /trends dashboard.
 * Session-gated, rate-limited 1/min/user (rule 16). Enqueues the same
 * ingest-trends BullMQ job as the cron route, with source='manual-admin'.
 *
 * Rule 14: no `jobs` DB row.
 */
export async function POST() {
  const session = await getServerSession();
  if (!session?.user) {
    return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
  }

  const gate = tryAcquireRefreshToken(session.user.id);
  if (!gate.allowed) {
    return NextResponse.json(
      { error: "rate_limited", retryAfterMs: gate.retryAfterMs },
      { status: 429, headers: { "Retry-After": String(Math.ceil(gate.retryAfterMs / 1000)) } }
    );
  }

  const [run] = await db
    .insert(trendIngestionRuns)
    .values({
      regionCode: "KR",
      status: "pending",
      source: "manual-admin",
    })
    .returning({ id: trendIngestionRuns.id });

  await getQueue().add("ingest-trends", {
    payload: {
      ingestionRunId: run.id,
      regionCode: "KR",
    },
  });

  return NextResponse.json(
    { ingestionRunId: run.id, source: "manual-admin" },
    { status: 202 }
  );
}
```

Note: Vercel Cron sends requests with method GET by default. Our route expects POST. Vercel Cron supports POST for hobby+ accounts via the `headers` config. However, in practice, Vercel Cron dispatches an HTTP request using the method the route handler accepts, AND Vercel also passes an `Authorization: Bearer <VERCEL_CRON_SECRET>` header for verification. BUT we require `x-cron-secret` per rule 11. Resolution:

- Export BOTH `GET` and `POST` handlers from `src/app/api/cron/trend-ingest/route.ts` that delegate to a shared `handle()` function. Vercel Cron hits GET by default; our own testing hits POST.
- Accept the secret from EITHER `x-cron-secret` header OR `Authorization: Bearer <CRON_SECRET>` header.

Update `src/app/api/cron/trend-ingest/route.ts` secret check accordingly:

```ts
function validateSecret(request: NextRequest): boolean {
  const header = request.headers.get("x-cron-secret");
  const bearer = request.headers.get("authorization")?.replace(/^Bearer\s+/i, "");
  const provided = header ?? bearer;
  return typeof provided === "string" && provided === env.CRON_SECRET;
}
```

and export both methods:

```ts
async function handle(request: NextRequest) {
  if (!validateSecret(request)) {
    return NextResponse.json({ error: "unauthorized" }, { status: 401 });
  }
  // ...existing body...
}

export async function POST(request: NextRequest) {
  return handle(request);
}
export async function GET(request: NextRequest) {
  return handle(request);
}
```

Document in vercel.json comments (JSON doesn't support comments; document in the plan's deployment runbook note below):

> Vercel Cron deployment note: set `CRON_SECRET` in Vercel dashboard (Environment Variables) to a 16+ char random string. Vercel Cron will dispatch via HTTPS using method GET. The route accepts both methods and validates the secret from `x-cron-secret` header (for manual testing) or `Authorization: Bearer` header (for Vercel's default cron auth). For Vercel Cron specifically, also set `CRON_SECRET` matching the `$CRON_SECRET` Vercel Cron injects on request.

## Risks & Mitigations

- **Risk**: Vercel Cron's request method or auth header shape changes and diverges from our implementation. **Mitigation**: route accepts both GET and POST and reads the secret from two header names (`x-cron-secret` and `Authorization: Bearer`). Both are exercised by tests.
- **Risk**: In-memory rate limiter resets on Vercel cold start, letting a user burst more than 1 req/min from the client's perspective. **Mitigation**: documented in the module header — acceptable for Phase 9. A Redis-backed replacement is a drop-in swap of `tryAcquireRefreshToken` when Phase 12 scales out.
- **Risk**: The cron route creates a `trend_ingestion_runs` row even when BullMQ enqueue fails, leaving an orphan pending row. **Mitigation**: the handler in 09-02 marks the run `failed` if it cannot find the row OR if BullMQ never delivers the job. Optionally: wrap the insert + enqueue in a try/catch that deletes the orphaned run row on enqueue failure — leave this as a follow-up.
- **Risk**: Without `CRON_SECRET` set in Vercel production env, the cron route 401's every call and ingestion never happens. **Mitigation**: the plan's deployment runbook note documents setting `CRON_SECRET`. The env var is required at build time via zod (`z.string().min(16)`), so Vercel builds will fail loudly if it's missing, catching the mistake early.
